live UIs for Rails · open source

Rails, live by default. Write normal ERB. Ship realtime apps.

Upkeep makes every server-rendered page reactive. Change a record; every browser viewing the views that depend on it updates in place. No broadcasts to wire up, no cache keys to maintain, no SPA to build. Rails apps move at chat-app speed with the productivity of plain ERB.

The whole surface is one line per reactive model. The rest of your app is the Rails you already write. Behind the scenes, Upkeep tracks what each render touched and, when data changes, pushes only the HTML that actually changed to only the clients that care. Targeted, fast, and small enough to reason about.

§ 02

How it works

five pieces · small, auditable, fast
01the compiler

Herb + ReActionView turn ERB into instrumented templates.

parse · walk · rewrite · at boot

Rails' default ERB handler is an opaque string concatenator. It produces output; it doesn't know what is an expression, what is a tag, what is a conditional. That makes dependency tracking impossible from the outside.

Herb is an HTML-aware ERB parser written in C with Ruby, JS, and WASM bindings. It produces a real AST: <h1> becomes an HTMLElementNode, <%= @message.title %> an ERBContentNode inside it. ReActionView plugs that parser into Rails as a template handler, replacing erubi for reactive partials.

Upkeep hooks into the compilation pipeline and rewrites the tree. Every <%= expr %> becomes Capture.capture(slot_index). Every <% if %> / <% elsif %> becomes Capture.branch. Every @ivar read becomes Capture.ivar. Reactive partials get wrapped in a <signal-fragment> boundary.

During a normal page render the Capture calls are zero-cost passthroughs. During a re-render they record a slot-by-slot decomposition of the fragment. Invisible to authors; you still see plain ERB.
02read-set tracking

The BatchBroadcasts middleware captures what each render touched.

predicates · type-cast · structured

When a request comes in, QueryTracking activates: an ActiveRecord::Relation prepend that intercepts every query method and captures its WHERE predicates as structured Predicate objects, one per condition: (table, column, operator, value, type).

Values go through ActiveRecord's type system, so the String "42" and the Integer 42 collapse to the same predicate. The render runs, and the middleware collects every predicate into a ReadSet:

ReadSet {
  (messages, room_id, =,  17),
  (users,    id,      IN, [3, 8, 12]),
  (rooms,    id,      =,  17),
}
The page is a materialized view; the ReadSet is its defining query. That single idea replaces every broadcast target, cache key, and touch: chain in your app.
03subscription & handoff

Rack env is captured with the read-set and stored.

replay-ready · per-subscriber · signed handoff

After the render, the middleware captures the rack env: cookies, session, Warden state, everything needed to replay the render with the same current_user. It stores the URL, ReadSet, fragment metadata (locals, partial paths, record reverse index), and slot states in the configured subscription store (:memory, :broker, or :active_record).

Before the response goes out, a <signal-stream-source> element is auto-injected with a signed token referencing the subscription. The browser parses it, opens an ActionCable connection, and subscribes to the per-subscription stream.

04the trigger

Invalidation is explicit per model. Routing is O(log N + k).

Kung-Robinson 1981 · Convex-style index

Reactivity is per-model and opt-in. A Broadcastable model's after_commit calls Upkeep.invalidate(record, event:, changes:). The payload includes both current and pre-change values, because a row that no longer matches a query still needs to evict itself from read-sets that previously captured it.

The InvalidationRouter picks a transport. Small deployments broadcast to a table-scoped ActionCable stream and each channel runs its own overlap check. Larger ones consult a shared SubscriptionIndex: a two-tier (table, column, value) hash for equality and IN, plus an always-check set for ranges and negations. Lookup is O(log N + k), the same data structure Convex uses in production.

Each receiving channel runs the overlap check: does the changed row satisfy any predicate in my read-set? If room_id changed from 17 to 22, two read-sets are affected: the one that matched room_id = 17 (losing the row) and the one that matches room_id = 22 (gaining it). The check is O(predicates), not O(rows).

If nothing overlaps, the channel does nothing. Most updates touch most subscriptions zero times. That is where the scale comes from.
05re-render, diff, morph

In-process render. Statics cached. Dynamics streamed.

per-fragment fast path · full-page fallback

On positive overlap the channel re-renders in-process, in the same Ruby process as the WebSocket. Two paths exist. Per-fragment runs only that partial through ActionView, restoring I18n locale, time zone, and Current attributes from the captured snapshot. Full-page runs an in-process Rack call through the captured rack env when the fragment touches state outside the partial.

The new render executes with SlotContext set, so the Capture calls now fire and emit a fresh statics / dynamics decomposition. The channel diffs against the stored slot state and ships only the changed dynamics:

# first push: statics + dynamics
{ "fragments": [{ "id": "card_42_frag_91eaf6fb",
                "s": ["<h1>", "</h1>"],
                "d": { "0": "Updated Title" } }] }

# subsequent pushes: dynamics only
{ "fragments": [{ "id": "card_42_frag_91eaf6fb",
                "d": { "0": "New Title" } }] }

The client merges changed dynamics with cached statics, reconstructs full HTML, and uses Idiomorph to morph the existing fragment in place. Focus, cursor, scroll, form state, and Stimulus instances survive the update. The UI never flickers.

§ 04

Six things server-side reactivity got wrong

and how we fixed each one

Live HTML from the server is an old idea. Previous attempts hit the same six walls. Each is a real engineering problem with a real solution. Here they are, with the fix shown next to each one.

Lesson01Per-client state that never stops growing
Then

Earlier systems kept a copy of every row each client had seen, so the server could diff new data against it. As clients queried more data, the per-client buffer grew without bound. Long-lived sessions eventually OOM'd the server.

Now, in Upkeep

Per-channel state is bounded structurally. We hold read-set predicates and a small delivery cap, not a shadow copy of the database. The upcoming result-hash refactor reduces this to a 32-byte hash per fragment per subscriber.

then: buffer grows · now: bounded
Lesson02Every write checked against every subscription
Then

When a row changed, the server asked every open subscription, “do you care about this?” With a thousand subscribers, one write became a thousand predicate checks. Throughput fell off a cliff long before CPU did.

Now, in Upkeep

A centralized router indexes subscriptions by (table, column, value). On a write, it looks up exactly the subscriptions that care, in O(log N + k). Ranges and negations fall into a small always-check set.

then: every sub checked · now: indexed lookup
Lesson03Everything reactive, nothing measurable
Then

Reactivity was on for every model and every query by default. When the server fell over, there was no way to see which view, which query, or which model was responsible. You could not profile what you could not name.

Now, in Upkeep

Reactivity is opt-in per model. Every step emits an ActiveSupport::Notifications event: upkeep.invalidation, upkeep.overlap_check, upkeep.batch_flush. You profile a live view the way you profile SQL.

then: opaque · now: instrumented
Lesson04No backpressure on bursts
Then

A bulk import emitted a write per row. Each write produced one push per affected subscriber. A reasonable import became millions of frames on the wire, and the first server to notice was the one that crashed.

Now, in Upkeep

Bursts are coalesced at every boundary. Per-channel queues merge duplicates, in-flight delivery is capped, a splay spreads fan-out, and Upkeep.batch collapses a thousand updates into one push per affected record on purpose.

then: firehose · now: coalesced
Lesson05Stateful connections, sticky sessions
Then

Each WebSocket held session state the server could not reproduce. Load balancers needed sticky sessions, deploys dropped live state, and horizontal scaling meant copying that state between nodes over the network.

Now, in Upkeep

The render path is a pure function of (url, fragment_id, locals, snapshot). The substrate for a stateless sidecar is already in place. Connections become cheap and portable.

then: sticky to one box · now: any worker
Lesson06Server-side rendering as the performance floor
Then

Every update re-ran a whole template on the server. Render cost compounded with fan-out. Teams migrated to clients-render-everything architectures to escape it.

Now, in Upkeep

Fragments render in roughly 0.3 ms in-process; full pages in 2.5 ms. The pipeline is stateless, so render workers scale horizontally. Statics are cached once; only the dynamics travel on each update.

then: one renderer · now: horizontal

The principles we hold

six invariants that hold at every scale

I
ReadSet is the source of truth

All invalidation decisions flow from predicate overlap, never from stream membership.

II
Rendering is a pure function

(url, fragment_id, locals, snapshot) → (statics, dynamics, read_set). No side effects, no connection state.

III
Delivery is targeted

The system knows which subscriptions are affected before sending. Broadcast-then-filter is the anti-pattern.

IV
Backpressure at every boundary

Per-channel queue, in-flight cap, drain limits, splaying, batch coalescing. Never unbounded.

V
Cost is visible

Every invalidation, render, and delivery is instrumented via ActiveSupport::Notifications.

VI
Reactivity is explicit

Models opt in. Columns can be filtered. Bulk operations can skip tracking.

§ 06

Built for fanout

one write, many viewers · the shape that breaks other systems

The hard part of live HTML is not rendering. It is routing. When one write has to reach N viewers, the system's scaling behavior is whatever it costs to decide who gets the update. We benchmark the two fanout shapes that cover most real apps.

Pattern A

Broadcast fanout

One write visible to a shared audience. Every viewer on a channel sees the same update. Typical of activity feeds, live scoreboards, shared timelines, and anything with a room model.

Pattern B

Graph fanout

Many writers, many viewers, each viewer seeing a different slice. Cards move, users drag, filters cross. Typical of kanbans, shared documents, CRMs, and dashboards where each pane shows a different query.

§ 07

The whole API, on one page

five methods · everything else is plain Rails
make a model live

One line. Every view that touches this model updates in real time across every connected browser. 

class Message < ApplicationRecord
  include Upkeep::Broadcastable
end
coalesce a burst

Thousand-row imports collapse to one push per affected record. Your users see one update, not a thousand flickers. 

Upkeep.batch do
  messages.each { |m| m.update!(status: "archived") }
end
batching across jobs

The same coalescing inside ActiveJob, Sidekiq, or a plain Thread. Background work stays invisible until it's done. 

Upkeep.batch_global do
  Message.find_each { |m| m.touch }
end
a nudge instead of a push

When you want the cheapest possible update, tell clients "something changed" and let them refetch via a normal HTTP GET. 

Upkeep.version_bump(@board)
stay silent

Migrations, backfills, audit logs, anything you don't want to push. The escape hatch is one block. 

Upkeep.skip_tracking do
  Message.update_all(status: "migrated")
end
§ 08

Research-grade foundations

incremental view maintenance for the web

Upkeep is a materialized-view engine for HTML. Incremental view maintenance and optimistic concurrency control are decades-old database techniques with a deep literature and well-understood performance. We took the shape that databases use to keep derived tables fresh and pointed it at the rendered page.

The page is a view. The read-set is its defining query. A write is a delta. The render is the row. Every decision in Upkeep follows from that analogy, which is why it holds up at scale.

The result is a system with strong theoretical guarantees about which fragments need to update, rather than a heuristic about which broadcasts might reach the right client.
